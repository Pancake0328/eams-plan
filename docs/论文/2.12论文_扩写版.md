## 摘要

企业资产管理贯穿采购、入库、领用、调拨、维修、盘点、报废等多个环节，涉及资产部、财务部、使用部门、维修部门等多角色协同。传统以 Excel 台账或分散系统记录资产信息的方式，容易产生“口径不一致、账实难核对、责任难追溯、盘点难闭环、维修过程不可视”等管理问题，进而导致资产闲置率高、重复采购、资产流失以及审计风险。

本文围绕“企业资产管理系统（Enterprise Asset Management System，EAMS）”的设计与实现展开研究与实践，基于实际业务场景，提出一套覆盖资产全生命周期的信息化解决方案。系统采用前后端分离架构：后端基于 **Spring Boot + MyBatis-Plus + Spring Security + JWT**，前端基于 **Vue 3 + TypeScript + Element Plus + Pinia + Vue Router**，并结合 ECharts 实现统计可视化。系统功能覆盖：组织与权限（RBAC）管理、资产分类与台账管理、采购单与入库、资产流转记录、生命周期阶段跟踪、盘点计划与盘点明细、报修审批与维修闭环、仪表盘统计等模块。

在安全与工程化方面，系统采用 JWT 无状态认证并在后端通过 `@PreAuthorize` 实现细粒度权限点校验，权限点来源于菜单/按钮配置（`sys_menu.permission_code`），从而保证“页面可见性控制”和“接口强制鉴权”一致；在数据一致性方面，对关键业务动作（入库、分配、调拨、报修、盘点完成等）以事务边界保证台账与历史记录一致写入；在可用性方面，对盘点明细等大数据量场景使用分页查询与分批提交，降低单次交互成本。

通过对核心业务流程的实现与验证，系统能够实现资产全生命周期的可追溯管理，并为企业提供可量化的统计分析与审计依据。本文的工作为中小企业基于开源技术栈快速落地资产管理平台提供了参考。

**关键词：** 企业资产管理；RBAC；JWT；Spring Boot；Vue 3；盘点管理；报修流程

# 绪论

## 研究背景及意义

### （1）研究背景

随着企业经营规模扩大、固定资产与低值易耗品数量增长，资产管理从“台账登记”逐步演变为需要精细化运营与审计合规支撑的管理活动。尤其在制造业、连锁企业与多部门组织中，资产分布广、流转频、维修工单多、盘点周期长，导致资产数据在部门之间天然呈现“碎片化”特点。

在许多企业的现实实践中，资产数据往往分散在多个载体：采购部门维护采购清单、财务部门维护折旧或金额台账、资产管理员维护固定资产编号，使用部门维护借用/领用记录，维修部门记录维修情况。这种分散记录带来三类常见问题：第一，数据不同步导致账实不符；第二，责任链条不清导致追责困难；第三，缺乏统计口径导致管理层难以决策。

### （2）研究意义

从企业治理角度，建设统一的资产管理平台具有显著意义：

- **提升资产透明度与可追溯性**：将资产编号、归属部门、责任人、状态与历史动作（入库/分配/调拨/送修/报废等）形成“单一事实来源（Single Source of Truth）”，为审计与责任认定提供依据。
- **降低管理成本与风险**：通过流程化、标准化减少重复录入与人工核对成本，降低资产丢失、闲置、重复采购、违规处置等风险。
- **提高盘点效率并形成闭环**：将盘点计划、盘点明细、差异记录与结果归档统一管理，支持按范围、按分类、抽样/专项等方式盘点，提高盘点执行效率与结果可复用性。
- **提升维修过程可视化能力**：将报修、审批、维修、完成的状态机固化到系统中，并与资产状态联动，为维修响应与成本控制提供数据支撑。
- **支撑管理决策**：通过统计分析模块按分类、状态、部门、时间维度展示资产结构与趋势，为采购预算、资产处置策略与运维投入提供依据。

为便于明确问题域，表 1-1 总结了传统管理方式的典型痛点与信息化系统的对应改进点。

| 序号 | 传统痛点 | 典型表现 | 信息化改进思路 |
|---|---|---|---|
| 1 | 台账分散 | 多部门多本账、字段口径不一 | 统一资产数据模型与编码规范 |
| 2 | 流转不可追溯 | 调拨/领用无完整记录 | 以“动作记录 + 生命周期记录”固化历史 |
| 3 | 盘点难闭环 | 盘点数据分散、差异处理难 | 盘点计划+明细+状态流转+结果归档 |
| 4 | 报修不可视 | 维修进度靠口头沟通 | 报修工单状态机与资产状态联动 |
| 5 | 决策支撑不足 | 无统计、无趋势、无异常预警 | 仪表盘与统计报表、指标体系 |

## 国内外研究现状

资产管理系统在国内外均有较为成熟的实践。国外产品通常强调与 ERP、财务系统深度集成，具备固定资产会计处理、折旧策略、资产评估等功能；同时更重视流程标准化与合规审计。国内资产管理产品更强调场景化落地（例如事业单位固定资产、制造业设备资产、IT 资产等）以及与组织权限、人员、部门的灵活配置。

从研究与工程实践角度看，当前资产管理系统的关注点主要体现在以下方面：

1. **全生命周期与可追溯性**：以状态机或阶段模型表示资产在“购入—入库—使用—维修—闲置—报废”等阶段的演化，通过历史记录支撑审计追踪。
2. **权限与安全控制**：采用 RBAC/ABAC 等模型控制不同角色对资产数据与操作动作的访问边界；同时结合日志审计、防越权、防重放等策略。
3. **数据标准化与数据治理**：通过统一分类、编码、数据字典与主数据管理（MDM）提升数据质量；在组织变更、人员变动时能保持历史一致性。
4. **可视化与分析决策**：将资产结构、状态分布、部门分布、维修成本、盘点差异等指标可视化，形成指标体系与管理闭环。
5. **集成与扩展能力**：与采购平台、财务系统、OA/审批系统、条码/二维码系统、移动端应用等集成，提高业务覆盖与现场效率。

本文的系统实现以“可落地、可扩展、可审计”为目标，采用企业常用的开源技术栈实现核心流程，并为后续扩展（财务折旧、移动扫码盘点、多组织数据权限等）预留空间。

# 相关技术基础

## 开发平台及工具

### 开发平台

本系统采用典型的企业级 Web 应用技术路线：后端使用 Java 17 与 Spring Boot 3.x，基于 Maven 进行依赖管理与构建；前端使用 Vue 3 + TypeScript，结合 Vite 构建工具实现开发热更新与生产打包。选择该平台的核心考虑包括：生态成熟、社区资源丰富、可维护性强、可扩展性好，以及与企业常见基础设施（MySQL、Redis、Nginx）兼容。

在开发流程方面，后端以 REST API 的方式提供服务，前端通过 Axios 访问并在请求头携带 JWT Token，实现无状态会话。前后端分离有利于职责解耦与团队协作，也更易进行灰度发布与独立扩容。

### 开发工具

为了保证工程质量与协作效率，本系统在开发阶段使用如下工具链：

- **IDE 与编辑器**：IntelliJ IDEA（后端）、VS Code/WebStorm（前端）。
- **版本控制**：Git，采用分支策略进行功能开发与代码合并。
- **接口调试与联调**：Swagger UI（自动生成接口文档并支持在线调试）、Postman。
- **数据库工具**：Navicat、DataGrip，用于 SQL 调试、数据检查与脚本管理。
- **构建与依赖管理**：Maven（后端）、npm（前端）。

在企业工程实践中，工具链的价值不仅是“提高效率”，更重要是形成可重复的构建与交付流程，从而保证项目在不同环境、不同人员参与下仍能稳定迭代。

## 软硬件环境

### 服务器硬件环境

EAMS 属于典型管理后台系统，核心压力来自列表分页查询、盘点明细更新与统计查询。建议生产环境最低配置：4 核 CPU、8GB 内存、100GB 磁盘。若盘点明细与历史记录长期累积，建议根据数据量增长定期归档，并将日志与数据库存储规划纳入容量评估。

### 客户端硬件环境

系统面向管理端用户，通过浏览器访问即可。客户端建议 8GB 内存以上，保证多标签页、复杂表格与图表渲染时的体验。

### 软件环境

- **操作系统**：Windows Server / Linux（推荐 Ubuntu/CentOS）。
- **运行时**：JDK 17。
- **数据库**：MySQL 8.x。
- **缓存**：Redis（可选，用于缓存或扩展场景）。
- **Web 服务器**：Nginx（用于前端静态资源托管与反向代理）。

对于企业部署而言，建议将配置与密钥放入环境变量或配置中心（本文项目采用配置文件方式，便于学习与演示）。

## 后端技术栈

### 核心框架：Spring Boot

Spring Boot 提供自动配置与“约定优于配置”的开发体验，能够快速搭建可运行的 Web 服务。对于企业资产系统而言，后端不仅需要提供 CRUD 接口，更需要处理事务、校验、权限与异常等横切关注点。Spring Boot 与 Spring MVC、Spring Security、AOP 等组件形成完整生态，适合构建可维护的业务系统。

### 持久层框架：MyBatis-Plus

MyBatis-Plus 在 MyBatis 基础上提供通用 CRUD、分页查询、条件构造器与代码生成等能力，可显著减少样板代码。在资产管理场景中，大量模块（资产、盘点、报修、采购）都包含“条件过滤 + 分页列表 + 详情 + 状态更新”的典型需求，MyBatis-Plus 的分页与条件构造能力能够提升实现效率。

### 数据库：MySQL

MySQL 8.x 具备成熟稳定的事务支持与索引机制，适合承载资产台账与历史记录等强一致性数据。系统在数据库设计上采用逻辑删除字段（`deleted`）与创建/更新时间字段，便于审计与数据归档；对资产编号、盘点编号、报修编号等业务唯一键建立唯一索引，保证并发下的数据正确性。

### 缓存与中间件：Redis（可选）

Redis 可用于缓存热点数据（例如菜单树、权限集、统计结果）或作为分布式锁、短期临时存储（验证码、一次性 token 等）。本文项目中 Redis 为可选组件，系统在不启用 Redis 时仍可正常运行；在企业落地时，可根据并发与数据量选择性启用。

### 安全框架：Spring Security + JWT

系统安全分为两个层次：

1. **认证（Authentication）**：用户登录后由后端签发 JWT，前端在后续请求中携带 `Authorization: Bearer <token>`；后端过滤器解析 token 并构建 `SecurityContext`。
2. **授权（Authorization）**：基于 RBAC 权限点模型，后端在方法级通过 `@PreAuthorize` 校验权限点，确保即使前端隐藏按钮也无法越权调用接口。

该方案具有无状态、可扩展、与前后端分离架构匹配的特点，适用于管理后台系统。

### 工具库

项目引入 Lombok 以减少 getter/setter 样板代码，引入 Hutool 等工具库简化日期、字符串等常用操作；若引入 MapStruct 可进一步提升 DTO/VO 转换的可维护性。

### API 文档：OpenAPI/Swagger

系统通过 SpringDoc OpenAPI 自动生成接口文档，并提供 Swagger UI 页面用于联调与验收。对于企业项目，接口文档的价值不仅是“开发时调试”，更是“交付时对齐需求与验收口径”的依据，因此建议在交付阶段导出 OpenAPI 文档作为基线。

### 其他组件

系统包含统一响应结构、统一异常处理、参数校验等工程化组件，保证接口返回一致、错误可追踪，并便于前端统一处理提示信息。

## 前端技术栈

### 核心框架：Vue 3 + TypeScript

Vue 3 的 Composition API 适合构建复杂页面，TypeScript 提升类型安全与可维护性。管理后台页面通常包含大量表格、表单与对话框组件，若缺乏类型约束容易产生字段不一致、接口参数错误等问题，因此 TypeScript 的引入能够降低长期维护成本。

### 构建工具：Vite

Vite 提供快速启动与热更新能力，并基于 ES Module 在开发阶段减少打包等待时间；在生产构建阶段输出静态资源，适合由 Nginx 部署。

### UI 组件库：Element Plus

Element Plus 提供成熟的后台组件体系，包括表格、分页、表单校验、树形组件、对话框等。资产管理系统的核心交互（筛选、编辑、批量操作、状态变更）均可用成熟组件实现，减少重复造轮子并保证一致的 UI 体验。

### 状态管理：Pinia

Pinia 用于管理用户信息、token、权限集合、菜单树与全局配置等。通过集中式状态管理可以减少跨组件传参，提高模块间协作效率。

### 路由管理：Vue Router

路由不仅用于页面导航，也承担权限控制的重要职责。系统采用 `meta.permission` 标识路由所需权限，配合路由守卫实现“未登录拦截”“无权限重定向到可访问页面”等策略，从而避免用户通过手动输入 URL 进入未授权页面。

### HTTP 客户端：Axios

Axios 通过请求拦截器统一注入 JWT token，通过响应拦截器统一处理 401/403 与业务错误码，从而保证前端在异常情况下能给出一致的提示与跳转行为。

### 数据可视化：ECharts

仪表盘模块使用 ECharts 展示资产状态分布、分类占比、部门资产数量等指标。可视化的核心是“指标定义 + 数据聚合 + 图表映射”，因此本文在统计接口与图表配置之间建立清晰的映射关系，便于扩展更多指标。

## 开发规范及约束

### 后端框架分层

后端遵循 Controller → Service → Mapper 分层：

- Controller 层负责参数接收、基础校验与权限注解；
- Service 层负责编排业务规则、处理事务边界、组织多表写入；
- Mapper 层负责数据库访问与复杂查询。

该分层有助于保持职责清晰，并便于单元测试与后续重构。

### DTO/VO 规范

系统使用 DTO/VO 承载输入输出，避免直接暴露数据库实体，降低接口与数据库结构耦合。尤其在资产列表展示与详情展示时，VO 可组合分类名称、部门名称等冗余字段，提高前端渲染效率。

### 统一响应格式

系统采用统一响应结构（如 `Result<T>`），包含状态码、消息与数据字段。统一响应的意义在于：前端可统一解析逻辑、异常可统一提示；后端可统一记录日志与定位问题。

### RESTful API 设计规范

接口设计遵循资源化与语义化原则：

- `GET` 用于查询；`POST` 用于创建；`PUT` 用于更新；`DELETE` 用于删除。
- 列表接口使用分页参数（如 `current/size`），避免全量返回导致性能问题。
- 重要动作类接口（如“开始盘点”“完成维修”）应具备明确动作语义并在后端进行状态校验。

### 命名与权限点规范

权限点建议采用 `模块:资源:动作` 的方式命名（例如 `asset:record:transfer`），便于角色授权与审计。数据库表、字段建议遵循下划线命名，保持一致性。

### 核心功能详细需求

#### 1）采购入库生成资产（UC-01）

- **参与者**：采购管理员、资产管理员。
- **前置条件**：采购单已创建，采购明细存在且处于“待入库/部分入库”状态。
- **主成功场景**：
  1. 采购管理员创建采购单，录入供应商、采购日期与采购明细（资产名称、分类、规格、数量、单价等）。
  2. 资产管理员在采购明细上执行“入库”操作，系统校验入库数量不超过采购数量。
  3. 系统根据入库数量生成对应数量的资产台账记录（`asset_info`），并写入入库类型的流转记录（`asset_record`）与生命周期记录（`asset_lifecycle`，阶段=购入/闲置）。
  4. 采购明细的已入库数量增加，采购单状态更新为“部分入库/已入库”。
- **异常场景**：入库数量非法、采购单已取消、并发重复入库导致超入库（需要后端校验与事务控制）。
- **输出**：新生成的资产编号列表、入库结果、采购状态变更。

#### 2）资产分配/调拨/归还（UC-02）

- **参与者**：资产管理员。
- **核心规则**：
  - 分配：闲置资产可分配到部门/责任人，状态变为“使用中”。
  - 调拨：使用中资产可从部门 A 调拨到部门 B，责任人与归属变化需记录。
  - 归还：使用中资产归还后变为“闲置”，并记录归还动作。
- **状态约束**：若资产处于“维修中”“报废”“取消采购”等状态，不允许执行分配/调拨等操作。
- **输出**：台账更新、流转记录追加、生命周期阶段更新。

#### 3）盘点计划创建与执行（UC-03）

盘点流程是资产管理的关键闭环，系统支持全面盘点、抽样盘点与专项盘点三种类型（与数据库 `asset_inventory.inventory_type` 一致）。

- **计划创建**：录入盘点名称、类型、计划日期范围；抽样盘点需录入抽样数量；专项盘点需选择分类。
- **盘点启动**：将盘点计划从“计划中”变更为“进行中”，生成盘点明细（`asset_inventory_detail`），并将应盘资产数量写入计划表。
- **盘点执行**：执行人按分页查询盘点明细，对每条资产录入盘点结果（正常/位置异常/状态异常/丢失等），并记录盘点人、时间与备注。
- **盘点完成**：系统汇总正常/异常数量，写回盘点计划表并锁定结果，防止重复修改。

报修模块采用状态机管理，数据库字段 `asset_repair.repair_status` 体现“待审批→已审批→维修中→已完成/已拒绝”的流程。

- **审批**：维修管理员审批报修，审批通过后可指定维修人员；拒绝则记录原因。

- **维修完成**：填写维修结果与费用，结束维修并将资产状态恢复为原状态或闲置状态，记录生命周期与流转历史。

### 业务数据对象与边界

系统围绕“资产”这一核心对象构建：

- `asset_info`：资产主数据（编号、名称、分类、金额、部门、责任人、状态等）。
- `asset_record`：资产动作历史（入库、分配、调拨、归还、报废、送修、维修完成等）。
- `asset_lifecycle`：资产生命周期阶段记录（购入、使用中、维修中、闲置、报废、取消采购等）。
- `asset_inventory` 与 `asset_inventory_detail`：盘点计划与盘点明细。
- `asset_repair`：报修工单记录。

这种“主数据 + 历史记录 + 状态机/阶段”的建模方式，能同时满足日常操作与审计追溯需求。

## 非功能需求

### 性能需求

结合管理后台特点与盘点大数据量场景，系统提出如下性能目标（可作为企业验收建议指标）：

1. **列表查询**：常规列表分页接口（资产列表、采购列表、报修列表）在数据量 10 万级时，单页查询（20 条）应控制在 500ms 以内（P95）。
2. **盘点明细**：盘点明细必须分页加载，禁止一次性全量返回；单页更新应支持批量提交并在 1s 内完成。
3. **权限加载**：用户登录后加载菜单树与权限集应在 1s 内完成（以保证首次进入系统体验）。

### 可靠性需求

- **事务一致性**：关键动作需保证“主表更新 + 历史记录写入”在同一事务中完成，避免出现台账更新成功但记录缺失，或记录存在但台账未更新的情况。
- **幂等性与并发控制**：对于入库、盘点完成等动作，需通过状态校验与数据库约束避免重复提交。
- **异常可追踪**：统一异常处理应返回可理解的错误信息，并在日志中包含定位所需的上下文（用户、接口、参数摘要）。

### 安全性需求

系统安全目标不仅是“能登录”，更是“可控、可审计、不可越权”。主要要求如下：

1. **身份认证安全**：密码加密存储（不可明文），token 过期机制明确；前端 token 存储需防止被轻易泄露。
2. **授权安全**：后端必须强制校验权限点（401/403），禁止仅依赖前端隐藏按钮。
3. **输入校验**：对关键字段进行参数校验与长度限制，降低注入与异常数据风险。
4. **审计与日志**：关键操作（报废、调拨、盘点完成、维修完成）应记录操作人、时间与结果，必要时可扩展操作日志表。

### 易用性需求

- 表单必填项明确标识并提供校验提示；
- 关键动作二次确认（删除、取消、报废、完成盘点）；
- 列表支持常用筛选条件并保留筛选状态；
- 错误提示信息明确（例如提示“无权限”“状态不允许”“入库数量超限”等）。

### 可维护性需求

- 模块化与分层结构清晰，便于扩展新模块（例如折旧、扫码盘点、移动端报修）。
- 权限点命名规范，且可通过菜单配置统一管理，避免在代码中散落字符串。
- 数据表包含 `create_time/update_time/deleted` 等通用字段，便于归档与审计。

## 系统结构设计

### 总体架构

系统采用前后端分离的三层架构：

- **表现层（前端管理端）**：负责页面展示、交互与权限驱动的菜单渲染。

前后端分离能够使“页面层”与“业务规则”解耦，降低耦合度；同时后端 API 可复用于移动端/第三方系统对接。

## 功能模块设计

### 模块划分

系统按业务域划分为：系统管理、资产管理、采购管理、盘点管理、报修管理、生命周期管理与仪表盘统计等模块。模块划分的原则是“高内聚、低耦合”：每个模块围绕清晰的业务对象与流程，并通过接口与共享基础能力（权限、用户、部门）协作。

### 模块间关系

```text
采购管理 ->(入库)-> 资产台账(asset_info)
资产台账 ->(动作)-> 流转记录(asset_record)
资产台账 ->(阶段)-> 生命周期(asset_lifecycle)
资产台账 -> 盘点计划/明细(asset_inventory/_detail)
资产台账 -> 报修工单(asset_repair) -> 资产状态联动
仪表盘 <- 汇总统计(资产/采购/盘点/报修)
RBAC -> 控制上述模块访问权限
```

### 核心模块详细设计

#### 资产流转模块

资产流转以 `asset_record.record_type` 表示动作类型（1 入库、2 分配、3 调拨、4 归还、5 报废、6 送修、7 维修完成、8 报修拒绝）。每次动作都应完成三件事：

1. 校验资产当前状态是否允许该动作；
2. 更新资产台账状态与归属字段；
3. 写入动作记录与生命周期记录，形成完整可追溯链。

这种设计保证“台账状态”反映当前状态，“动作记录/生命周期”反映历史变化，避免历史被覆盖。

#### 盘点模块

盘点模块由计划表与明细表构成：计划表保存盘点类型、计划日期与汇总统计字段；明细表保存每件资产的盘点结果。盘点执行时必须采用分页加载明细，避免一次性加载导致性能问题。盘点完成时对结果进行汇总并锁定，形成审计可追溯的盘点档案。

#### 报修模块

报修模块采用状态机管理，审批与维修动作与资产状态联动：当报修进入“维修中”时，资产状态变更为“维修中”；当维修完成时，资产状态恢复并写入“维修完成”动作记录。报修记录还包含维修费用与维修结果字段，为后续成本统计提供基础。

## 数据库设计

### 数据库概述

数据库采用 MySQL，核心目标是保证一致性、可追溯性与可扩展性。系统采用逻辑删除字段 `deleted`，并在多数表中包含 `create_time/update_time` 字段，便于审计与数据恢复。关键业务编号（资产编号、盘点编号、报修编号、采购单号）建立唯一约束，保证数据唯一性。

#### （3）资产流转记录表 `asset_record`

该表记录资产每次动作的变更前后信息，包括部门、责任人、状态等。核心字段如下：

#### （5）盘点计划表 `asset_inventory` 与盘点明细表 `asset_inventory_detail`

盘点计划表保存盘点基本信息、计划日期、状态、汇总数量（应盘/实盘/正常/异常）。盘点明细表保存每个资产的盘点结果与盘点人/时间。盘点结果枚举与数据库字段一致：1 未盘点、2 正常、3 位置异常、4 状态异常、5 丢失。

#### （6）报修记录表 `asset_repair`

报修表保存报修编号、资产信息、故障描述、优先级、审批/维修过程字段以及维修费用与结果。通过该表可实现维修过程可视化，并与资产状态联动。

# 系统实现

## 后端实现

### 项目结构

后端基于 Spring Boot 采用分层结构组织代码，通常包含：

- `controller`：对外 REST 接口，承载权限注解与参数接收；
- `service`：业务编排与事务控制；
- `mapper`：数据库访问；
- `entity/dto/vo`：数据模型；
- `security`：JWT 解析、认证授权配置；
- `common`：统一响应、异常处理、工具类等。

该结构便于模块化扩展（例如后续加入折旧模块只需新增对应包与接口）。

### 核心代码实现（说明）

后端实现的关键在于将业务规则固化在 Service 层，并通过事务保证多表一致写入。以“资产调拨”为例，典型实现步骤如下：

1. 校验请求参数与权限；
2. 查询资产当前状态与归属；
3. 校验状态允许调拨；
4. 更新 `asset_info.department_id/custodian/asset_status`；
5. 写入 `asset_record`（记录 from/to 与 old/new 状态）；
6. 写入 `asset_lifecycle`（阶段变化或归属变化摘要）。

### 权限控制实现

系统权限控制采用“菜单/按钮配置驱动”的 RBAC 模型：

- 菜单与按钮在 `sys_menu` 表配置，按钮类型为 `BUTTON`，并包含 `permission_code`（如 `asset:record:transfer`）。
- 用户与角色关联，角色与菜单/按钮关联，从而得到用户权限点集合。
- 登录后后端可返回用户权限集合，前端用于页面与按钮控制；同时后端在接口上使用 `@PreAuthorize` 强制校验。

后端鉴权流程如下：

```text
登录 -> 签发JWT(token中携带用户标识)
请求API -> 过滤器解析JWT -> 构造Authentication(authorities)
进入Controller方法 -> @PreAuthorize校验 -> 通过/403
```

为了防止“前端隐藏按钮但接口仍可调用”的越权风险，后端校验是最终安全边界。

## 关键技术实现

### 资产自动编号生成

资产编号是资产管理的核心标识，需满足“唯一、可读、可追溯”。结合项目示例数据（如 `YZ-20260208-0001`），本文给出一种编号规则：

- **分类前缀**：来源于资产分类编码（`asset_category.category_code`），如 `YZ`（椅子）。
- **日期段**：如 `YYYYMMDD`，表示入库日期。
- **序列号**：同一分类同一日期下递增序号，固定长度补零。

```text
示例：YZ-20260208-0001
  YZ      -> 分类编码
  20260208-> 日期
  0001    -> 当日序列
```

实现要点：

1. **并发安全**：序列生成需在事务中完成，并通过数据库唯一索引保证最终唯一性。
2. **可扩展性**：若后续需要按组织/仓库维度分段，可在前缀中加入组织编码。
3. **可读性**：编号应便于人工识别与扫码打印。

【截图：入库生成资产编号列表】

### 盘点明细分页查询与批量录入

盘点明细在资产规模较大时数据量可达数万条，若一次性加载将导致接口超时与浏览器卡顿。因此系统采用分页策略：

- 后端提供按 `inventory_id + page` 查询明细的接口；
- 前端表格使用分页组件，支持快速翻页；
- 录入盘点结果时支持批量保存当前页变更，避免一次提交全量数据。

该方案不仅提升性能，也提升用户体验：盘点人员可按区域/资产编号分批处理，减少一次性工作量。

【截图：盘点明细分页录入】

### ECharts 图表实现与指标口径

仪表盘实现的关键不是“画图”，而是指标口径定义。系统以数据库状态字段为统计维度：

- `asset_info.asset_status`：资产状态分布（闲置/使用中/维修中/报废）。
- `asset_category`：分类占比。
- `department_id`：部门资产数量与金额汇总。

后端提供聚合接口返回结构化数据，前端将其映射到 ECharts option。为保证统计一致性，应在文档中明确：统计是否包含逻辑删除数据、是否包含取消采购资产、按哪个时间字段统计等。

# 总结与展望

本文围绕企业资产管理系统 EAMS 的设计与实现开展了需求分析、系统设计、工程实现与测试验证工作。系统实现了资产全生命周期管理的核心能力：采购入库生成资产、资产台账维护、流转追溯、盘点闭环、报修审批与维修闭环，并通过仪表盘可视化为管理决策提供数据支撑。

在系统实现过程中，本文重点解决了以下工程问题：

1. **权限体系一致性**：通过“菜单/按钮配置驱动”的权限点体系，实现前端可见性控制与后端强制鉴权一致，降低越权风险。
2. **可追溯性建模**：通过 `asset_record` 与 `asset_lifecycle` 双历史表实现动作追溯与阶段追溯，保证审计可用。
3. **大数据量交互优化**：对盘点明细采用分页与分批保存策略，提升性能与用户体验。

同时，系统仍存在可进一步优化之处：例如财务折旧与会计口径、扫码盘点与移动端、跨组织数据权限与审计日志增强等。未来工作可从以下方向扩展：

- **财务与折旧**：引入折旧策略、会计期间、凭证接口，对接财务系统，实现资产账务一体化。
- **移动端与扫码**：支持二维码/条码打印与扫码盘点、移动报修，提高现场效率。
- **数据权限与审计增强**：增加部门数据权限、操作日志、异常告警与风险控制。
- **统计指标扩展**：增加维修成本趋势、资产利用率、盘点差异率等指标，并支持导出报表。
