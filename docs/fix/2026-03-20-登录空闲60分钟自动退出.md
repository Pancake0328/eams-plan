# 2026-03-20 登录空闲 60 分钟自动退出

## 1. 需求说明

将登录超时规则调整为：

- 用户连续 **60 分钟无操作** 自动退出登录；
- 用户有操作时，刷新超时时间（滑动窗口）。

---

## 2. 实现方案

本次采用前端统一会话空闲检测方案，避免分散到各页面重复实现。

### 2.1 核心策略

1. 登录后启动空闲监测；
2. 监听用户活动事件（点击、键盘、鼠标移动、滚动、触摸）；
3. 每次活动更新最后活跃时间；
4. 定时检查是否超过 60 分钟；
5. 超时后清理登录态并跳转登录页。

### 2.2 数据存储

使用 `localStorage` 记录最后活跃时间：

- `session:last-active-time`

说明：

- 记录在 `localStorage` 可覆盖页面刷新场景，刷新后仍可继续按真实空闲时长计算。

### 2.3 关键参数

- 空闲超时：`60 * 60 * 1000`（60 分钟）
- 检查间隔：`30 * 1000`（30 秒）
- 活跃时间写入节流：`5 * 1000`（5 秒）

---

## 3. 代码变更

### 3.1 `frontend/src/stores/user.ts`

新增并整合：

1. 空闲监测常量与事件列表；
2. `persistLastActiveAt`：写入最后活跃时间；
3. `startInactivityMonitor` / `stopInactivityMonitor`：登录态生命周期控制；
4. `checkInactivityTimeout`：超时判定与自动退出；
5. `handleVisibilityChange`：标签页回到前台时立即校验；
6. 登录成功后自动启动监测，登出后停止监测并清理数据；
7. 页面刷新但仍有 token 时自动恢复监测。

### 3.2 `frontend/src/utils/request.ts`

调整请求拦截与 401 处理：

1. 请求携带 token 时调用 `userStore.persistLastActiveAt()`，确保接口调用也计入“有操作”；
2. 401 时改为调用 `userStore.logout()` 统一清理逻辑（不再手工移除 token/userInfo）。

---

## 4. 行为结果

1. 用户持续有操作（页面交互或接口请求）时，会话不断续期；
2. 用户连续 60 分钟无操作时，系统自动登出并提示“登录超时，已自动退出，请重新登录”；
3. 手动登出与 401 异常登出都走同一套清理逻辑，行为一致。

3. 登录后仅触发接口请求，也应刷新活跃时间；
4. 超时后访问受保护页面应跳转登录页。
