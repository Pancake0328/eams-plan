## 1. 引言

### 1.1 编写目的
本说明书用于描述 EAMS 的总体架构、模块划分、数据设计、接口设计、安全设计、部署与运维设计等关键技术方案，作为开发实现、代码评审、测试与运维交付的依据。

### 1.2 设计原则
- **分层解耦**：Controller/Service/Mapper 分层，DTO/VO 分离。
- **安全优先**：后端强制鉴权与授权，前端仅做展示控制。
- **统一规范**：统一响应、统一异常、统一日志、统一权限命名。
- **可扩展性**：以模块化方式支持后续引入折旧、外部系统对接、移动端等。

## 3. 后端设计

### 3.1 代码组织与分层
以 `com.eams` 为根包，按业务域分模块：

- `system`：用户/角色/部门/菜单权限
- `asset`：资产信息/分类/流转记录
- `purchase`：采购单/入库/统计
- `lifecycle`：生命周期/盘点/报修（部分子模块）
- `dashboard`：统计聚合
- `common`：通用配置、安全、统一响应、异常、工具

典型分层职责：
- **Controller**：参数校验、权限注解、协议适配（DTO/Query）
- **Service**：业务编排、事务边界、规则校验
- **Mapper**：数据访问（MyBatis-Plus）
- **Entity/DO**：数据库映射对象
- **DTO/VO**：输入/输出对象，避免直接暴露实体

### 3.2 统一响应与异常
- 建议采用 `Result<T>` 统一返回：包含 `code/message/data`（以实际实现为准）。
- 全局异常处理：将业务异常、参数异常、权限异常统一映射为可读错误码与信息。
- 日志：关键操作写入业务日志或审计字段（operator、time）。

### 3.3 鉴权与授权设计（Spring Security + JWT）

#### 3.3.1 JWT 鉴权流程

```text
客户端登录 -> 后端签发 JWT
后续请求带 Authorization: Bearer <token>
  -> JwtAuthenticationFilter 解析 token
  -> 校验有效期/签名/用户信息
  -> 写入 SecurityContext
  -> @PreAuthorize 进行权限表达式校验
```

关键实现要点（摘要）：
- token 前缀：`Bearer `
- 有效期：`application.yml` 中 `jwt.expiration`（当前为 1800 秒）
- 认证失败返回 401，授权失败返回 403。

#### 3.3.2 RBAC 权限模型
- **用户（sys_user）**：系统主体
- **角色（sys_role）**：权限集合
- **菜单/权限（sys_menu）**：权限点载体（DIR/MENU/BUTTON）
- 关联关系：`sys_user_role`、`sys_role_menu`
- 权限标识：`sys_menu.permission_code`

后端授权方式：
- Controller 层使用 `@PreAuthorize("hasAuthority('xxx')")` 或等价表达式；
- 服务端为最终裁决，前端仅控制显示与交互。

### 3.4 业务模块设计（后端）

#### 3.4.1 资产模块（asset）
- **资产分类**：树形维护，约束删除（存在子节点/资产则不可删）。
- **资产信息**：台账查询、详情、编辑；关键变更建议通过“流转”完成。
- **资产流转**：入库、分配、调拨、归还、送修、维修完成、报废；
  - 每次流转写入 `asset_record`（或同名表）形成历史；
  - 同步更新资产当前状态（asset_info.status）与归属字段；
  - 同步写入生命周期阶段（asset_lifecycle）。

#### 3.4.2 采购模块（purchase）
- 采购单（头表）+ 明细表结构；
- 到货后按明细入库：生成资产与入库记录；
- 财务统计：按月/年生成账单、统计采购金额分布与趋势（权限为 `finance:*`）。

#### 3.4.3 盘点模块（inventory）
- 盘点计划支持多类型（全面/抽样/专项）；
- 启动盘点后生成盘点明细（或在执行中按需生成）；
- 明细分页查询用于支撑大数据量情况下的前端展示；
- 完成盘点后锁定结果并汇总差异。

#### 3.4.4 报修模块（repair）
- 报修单状态机：创建 → 审批（通过/拒绝）→ 开始维修 → 完成维修；
- 与资产状态联动：送修/维修完成动作保持一致；
- 关键节点写入历史，保证审计追溯。

#### 3.4.5 生命周期模块（lifecycle）
- 生命周期记录主要由业务动作联动写入；
- 系统提供查询与分页，不提供手工篡改入口（避免数据失真）。

### 3.5 关键技术点设计

#### 3.5.1 资产自动编号生成
设计目标：保证编码**唯一性、可读性、可追溯性**，并支持并发场景。
- 编码建议格式（示例）：`AST-YYYYMM-000001` 或 `分类前缀-日期-序列`。
- 序列生成建议：
  - 数据库序列表（如 `asset_number_sequence`）按日期/分类维护当前值；
  - 通过事务 + 行级锁/乐观锁保证并发安全；
  - 生成后落库并返回。

#### 3.5.2 事务一致性
以下操作建议使用事务：
- 采购入库（写入采购状态 + 生成资产 + 写入流转/生命周期）；
- 流转动作（更新资产 + 写入流转 + 写入生命周期）；
- 报修审批/完成（更新报修单 + 更新资产状态 + 写入记录）。

#### 3.5.3 数据权限（可扩展）
当前实现重点为功能权限（菜单/按钮/接口权限）。如需进一步限制“部门数据可见范围”，可在查询层引入数据权限：
- 按用户所属部门过滤资产数据；
- 管理员角色可查看全量；
- 通过 AOP/拦截器在 Mapper 层统一拼接过滤条件。

## 4. 前端设计

### 4.1 项目结构与路由
前端（`frontend/`）使用 Vue Router 管理路由，页面组件位于 `views/`，全局状态使用 Pinia。

- 登录成功后初始化：
  1) 用户信息；
  2) 权限集合（permission codes）；
  3) 菜单树（用于左侧导航）。

路由守卫设计要点：
- 未登录访问受保护路由：重定向到登录页；
- 已登录但未加载权限：先加载权限再放行；
- 路由 `meta.permission` 存在时，若无权限则重定向到“首个可访问页面”。

### 4.2 菜单渲染与按钮权限
- 菜单树由后端返回（树结构），前端递归渲染。
- 按钮级权限通过自定义指令 `v-permission` 控制：
  - 用法：`v-permission="'system:user:add'"` 或 `v-permission="['a','b']"`。
  - 原则：前端隐藏按钮仅提升体验，不替代后端授权。

### 4.3 请求封装与错误处理
- Axios 统一封装：
  - 请求拦截器添加 token；
  - 响应拦截器处理错误码（401 自动跳转登录/提示）。
- 表单校验与提交：统一 loading、错误提示与确认弹窗。

### 4.4 可视化设计
- 仪表盘采用 ECharts 绘制饼图/柱状图/折线图；
- 建议将图表数据获取与渲染解耦：
  - store 或 composable 负责拉取数据；
  - 组件负责渲染与自适应。

## 5. 数据库设计

### 5.1 设计说明
数据库以 MySQL 8 为主，包含 RBAC 基础表与业务表（资产、采购、盘点、报修、生命周期等）。脚本参考：`sql\\预交付脚本\\production_schema.sql`。

### 5.2 核心表（摘要）

> 说明：本节给出“企业交付层面”的核心字段摘要；完整字段以脚本为准。

#### 5.2.1 RBAC
- `sys_user`：用户
- `sys_role`：角色
- `sys_menu`：菜单/按钮权限（permission_code）
- `sys_user_role`：用户-角色
- `sys_role_menu`：角色-菜单

#### 5.2.2 资产域
- `asset_category`：资产分类树
- `asset_info`：资产台账
- `asset_record`：流转记录
- `asset_lifecycle`：生命周期

#### 5.2.3 盘点域
- `asset_inventory`：盘点计划
- `asset_inventory_detail`：盘点明细

#### 5.2.4 报修域
- `asset_repair`：报修单

#### 5.2.5 采购域
- `asset_purchase_order`：采购单
- `asset_purchase_detail`：采购明细

### 5.3 关系设计（ASCII 示意）

```text
sys_user 1---* sys_user_role *---1 sys_role 1---* sys_role_menu *---1 sys_menu

asset_category 1---* asset_info
asset_info 1---* asset_record
asset_info 1---* asset_lifecycle
asset_info 1---* asset_repair

asset_inventory 1---* asset_inventory_detail *---1 asset_info

asset_purchase_order 1---* asset_purchase_detail
asset_purchase_detail --(入库生成)--> asset_info
```

### 5.4 索引与约束建议
- `asset_info.asset_code` 唯一索引；
- 历史表（record/lifecycle/repair/inventory_detail）建议按 `asset_id`、`create_time` 建索引；
- 关联字段（role_id/user_id/menu_id）建索引保障权限加载性能。

## 6. 接口设计

### 6.1 RESTful 规范
- 使用 HTTP 动词表达语义：GET 查询、POST 新增、PUT 更新、DELETE 删除。
- URL 资源化命名，分页使用 `current/size` 或 `page/size`（以实现为准）。

### 6.2 鉴权与状态码
- 401：未认证/Token 失效
- 403：已认证但无权限
- 4xx：参数/业务校验
- 5xx：服务端异常

### 6.3 核心接口清单（摘要）
为避免文档与代码频繁漂移，建议以 Swagger 为准；以下列出关键模块常用接口类别：
- `/auth/*`：登录、登出、获取用户信息
- `/permission/user/{id}/*`：权限集合、菜单树
- `/users`、`/departments`、`/role`、`/menus`：系统管理
- `/asset/*`、`/asset/category/*`、`/asset/record/*`：资产域
- `/inventory/*`：盘点域
- `/repair/*`：报修域
- `/purchase/*`、`/purchase/statistics/*`：采购与统计

> 如需“可验收的全量接口列表”，推荐在交付时从 Swagger 导出 OpenAPI JSON 作为附件。

## 7. 部署与运维设计

### 7.1 配置管理
- `src/main/resources/application.yml`：端口、context-path、DB、Redis、JWT 等。
- 建议生产环境通过环境变量/外部配置覆盖敏感信息（DB 密码、JWT 密钥）。

### 7.2 反向代理与跨域
- 生产建议使用 Nginx 统一入口并做 `/api` 反向代理；
- 若前后端分离部署，需配置 CORS 允许来源与 Header（Authorization）。

### 7.3 数据备份与恢复
- MySQL：每日全量备份 + binlog 增量；
- 备份校验：定期演练恢复；
- 关键表（资产台账、流转、盘点、报修、采购）需保证备份策略。

### 7.4 日志与监控（建议）
- 日志：按模块输出 INFO/ERROR，关键操作记录操作人、资产编号等；
- 监控：JVM、接口耗时、DB 连接池、Redis；
- 告警：接口 5xx、登录失败异常频率、慢查询。

## 8. 安全设计补充

### 8.1 常见安全风险与对策
- **越权访问**：后端 `@PreAuthorize` 强校验；
- **SQL 注入**：ORM 参数化，禁止拼接 SQL；
- **XSS**：输入校验、输出转义；
- **暴力破解**：可引入登录失败次数限制/验证码（后续增强）。

### 8.2 审计与合规（建议）
- 资产关键动作记录：入库/分配/调拨/报废/盘点完成/报修审批等；
- 可扩展审计日志表或对接 ELK。
